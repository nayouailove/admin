# 학생 채팅 관리자 페이지

선생님이 담당 학생 계정을 등록하고, 등록된 학생의 기존 채팅 목록으로 이동할 수 있는 관리자 페이지입니다.

## 핵심 기능

- 학생 ID 또는 학생 이름으로 학생을 등록할 수 있습니다.
- 등록된 학생 목록을 조회할 수 있습니다.
- 등록된 학생 수를 화면에서 확인할 수 있습니다.
- 학생 이름 또는 이동 아이콘을 누르면 기존 서비스의 채팅 목록 페이지로 이동합니다.
- 잘못 등록한 학생은 삭제할 수 있습니다.
- 학생 목록은 한 페이지에 7명씩 보여주며 페이지 이동이 가능합니다.

## 화면 흐름

1. 선생님이 관리자 페이지에 접속합니다. 
2. 학생 ID 또는 학생 이름을 입력해 학생을 등록합니다.
3. 등록된 학생 목록에서 학생 이름 또는 이동 아이콘을 클릭합니다.
4. 기존 채팅 목록 페이지로 이동합니다.

채팅 목록 이동 주소는 아래 형식을 사용합니다.

```text
https://edu.telliot.co.kr/device-chat-list/{학생ID}
```

## 로그인 처리 방식

현재 과제에서는 기존 로그인 서비스와 아직 직접 연동하지 않았습니다.

그래서 선생님 로그인 상태를 임시로 시뮬레이션하기 위해 `.env`에 개발용 선생님 정보를 넣어 사용합니다.

```env
DEV_TEACHER_ID=teacher01
DEV_TEACHER_NAME=김선생
```

백엔드는 `GET /me` API에서 이 값을 읽어 현재 로그인한 선생님처럼 응답합니다.

실제 로그인 서비스와 연동되면 아래 방식으로 바꾸면 됩니다.

- 프론트엔드는 별도로 선생님 ID를 입력하지 않습니다.
- 백엔드는 쿠키, 세션, 또는 토큰을 검증합니다.
- 검증된 로그인 정보에서 선생님 ID를 꺼냅니다.
- `DEV_TEACHER_ID` 대신 실제 로그인한 선생님 ID로 학생 목록을 조회합니다.

즉, 지금은 아래 흐름입니다.

```text
.env의 DEV_TEACHER_ID
  -> 현재 선생님으로 간주
  -> 해당 선생님의 학생 목록 조회
```

나중에는 아래 흐름으로 바뀝니다.

```text
로그인 쿠키/세션/토큰
  -> 백엔드에서 검증
  -> 실제 선생님 ID 확인
  -> 해당 선생님의 학생 목록 조회
```

## 등록 로직

학생은 ID 또는 이름으로 등록할 수 있습니다.

다만 이름은 중복될 수 있기 때문에, 이름으로 검색했을 때 같은 이름의 학생이 여러 명이면 바로 등록하지 않습니다.

```text
학생 ID로 등록
  -> 해당 ID를 가진 학생 계정이 있는지 확인
  -> 있으면 등록
  -> 없으면 에러

학생 이름으로 등록
  -> 해당 이름의 학생 계정을 검색
  -> 1명만 있으면 등록
  -> 0명이면 에러
  -> 2명 이상이면 동명이인 에러
```

## 데이터 구조

현재는 `users` 테이블과 `teacher_students` 테이블을 사용합니다.

`users` 테이블은 실제 계정 목록입니다.

```text
학생 계정
선생님 계정 
(role로 구분)
```

`teacher_students` 테이블은 선생님이 어떤 학생을 등록했는지 저장하는 연결 테이블입니다.

```text
teacher01 선생님
  -> test10 학생 등록
  -> test100 학생 등록
```

이렇게 나눈 이유는 학생 계정 자체와, 선생님이 등록한 학생 목록은 서로 다른 정보이기 때문입니다.

## 사용 기술

- Frontend: React, Vite
- Backend: FastAPI
- Database: PostgreSQL
- Local Dev: Docker Compose
- Deploy Plan: AWS RDS, ECR, ECS Fargate, ALB, S3, CloudFront, Route 53, GitHub Actions

## 로컬 실행

```powershell
docker compose up --build
```

실행 후 접속 주소:

```text
프론트엔드: http://localhost:5173
백엔드 API 문서: http://localhost:8000/docs
```


## 배포

1. 로컬에서 Docker Compose로 전체 실행 확인
2. GitHub에 코드 업로드
3. RDS PostgreSQL 생성
4. FastAPI Docker 이미지를 ECR에 업로드
5. ECS Fargate와 ALB로 백엔드 배포
6. React 빌드 파일을 S3와 CloudFront로 배포
7. 필요하면 Route 53으로 도메인 연결
8. 마지막에 GitHub Actions로 자동 배포 구성
